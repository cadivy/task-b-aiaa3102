# 数据集取证与标签审计报告

**Topic B — Dataset Forensics and Label Audit**

**数据集**：UCI SMS Spam Collection，5,574 条英文短信，按课程提供的固定 manifest 划分为 train 4,460 / heldout 1,114
**任务**：审计该数据集的质量，定位六类可疑样本，并评估数据质量如何改变模型评估结果的含义

---

## 1. 摘要

本报告对一个由公开 UCI 语料构建的短信垃圾分类数据集进行了系统性取证审计，覆盖 handout 规定的六个审计目标，并比较了九种数据干预。核心结论有四点。

**第一，99.10% 的 held-out 准确率高估了任务的真实难度。** 63 个 held-out 行（占 5.7%）可从训练集恢复，参考基线在这些行上全部预测正确；将其移出评分后，spam 召回率从 93.96% 降至 90.91%。

**第二，浅层表面特征即可解释绝大部分性能。** 仅使用联系方式与促销类手工特征、完全不提供词序的逻辑回归探针达到 98.11% 准确率与 92.78% spam F1，与完整词模型的 99.10% / 96.55% 相差约 1 个百分点。数据集存在强 shortcut：模型无需理解语义即可获得接近上限的分数。

**第三，标注约定已成为该基准的主要噪声来源。** 参考基线在 held-out 上的 10 个错误中，7 个是真实的模型错误，另外 3 个是数据问题（2 个疑似标签错误、1 个标注政策模糊）。在数据保持现状的前提下，任何模型的 held-out 准确率上界为 99.73%；而当前分数之上的全部剩余空间仅 7 行，与存在标注争议的 3 行处于同一量级。

**第四，准确率会系统性掩盖不同数据问题的差异。** 九种干预对准确率的影响全部落在 0.7 个百分点以内，而 spam 召回率的跨度达 4.95 个百分点（90.91%–95.86%），且作用机制互不相同——遮蔽类干预只影响精确率，删行类只影响召回率。在 13.4% 的类别不平衡下，仅报告准确率会把机制迥异的数据缺陷压缩成同一个"几乎无影响"的假象。

审计全程遵循一条统一原则：**自动信号仅用于检索候选，最终认定必须叠加人工政策判断**。我们在每一类目标中都明确记录了被拒候选，并主动保留了刻意标记的 false positive，以证明发现集合经过校准而非"宁滥勿缺"。

## 2. 数据与方法

### 2.1 数据来源与完整性

原始语料为 UCI SMS Spam Collection，通过脚本自动下载并校验。归档文件 SHA-256 为 `1587ea43…3ce3`，解压后的原始文件 SHA-256 为 `7d039a24…c239d`。与课程固定 manifest 的连接保留官方划分，不重新生成任何行的归属。

标准工作表 `data/canonical_sms.csv` 通过 `load_canonical()` 读取，显式设置 `keep_default_na=False`，以避免 pandas 默认行为把短信正文中字面出现的 `NA`、`null`、`nan` 静默转换为缺失值——这些字符串在语料中确实存在，是有效文本而非缺失标记。

| 完整性检查      |          结果 |
| --------------- | ------------: |
| 总行数          |         5,574 |
| 训练行 / 保留行 | 4,460 / 1,114 |
| 唯一 ID 数      |         5,574 |
| 唯一 UCI 行号数 |         5,574 |
| 空文本行        |             0 |
| 解码编码        |         utf-8 |

### 2.2 数据画像

**类别分布。** spam 在全语料占 13.40%，train 与 heldout 的 spam 比例分别为 13.41% 与 13.38%，标签先验几乎完全一致，固定划分未出现可见偏移。

| 划分    |   ham | spam |  合计 | spam 占比 |
| ------- | ----: | ---: | ----: | --------: |
| train   | 3,862 |  598 | 4,460 |    13.41% |
| heldout |   965 |  149 | 1,114 |    13.38% |
| 全体    | 4,827 |  747 | 5,574 |    13.40% |

强类别不平衡是后文所有指标讨论的前提：一个恒定输出 ham 的分类器即可获得 86.62% 准确率。因此本报告全程以 **spam 召回率、spam F1 与 macro-F1** 为主要判据，准确率仅作参考。

**表面线索分布。** spam 与 ham 在长度、数字、联系方式与促销词上呈现极大差异：

| 指标            |   ham |   spam |
| --------------- | ----: | -----: |
| 字符数中位数    |    52 |    149 |
| 数字个数中位数  |     0 |     16 |
| 含 URL          | 0.29% | 18.47% |
| 含电话号形式串  | 0.10% | 60.78% |
| 含货币/资费线索 | 0.62% | 38.15% |
| 含促销词        | 9.16% | 87.55% |

这是描述性事实，本身并不构成"标签有问题"的主张，但它直接指向目标 5 的核心问题：这些线索是否解释了过多性能。

**划分可比性。** 在标签之外的表面指标上，train 与 heldout 高度相似（字符数中位数 62 / 60，URL 率 2.69% / 2.87%，促销词率 19.91% / 18.67%）。最明显的差异是 heldout 的电话号形式串比例略高（8.07% / 8.89%），这一点在解读 spam 召回率时应予考虑。

**语料层残留。** 按 UCI 原始行号分十档统计，spam 比例在 11.67%–15.80% 之间波动，未见按标签排序的痕迹。另有 309 行（5.54%）含 HTML 转义序列、483 行（8.67%）含非 ASCII 字符。这些残留予以保留而非清洗，以保证后续所有审计针对的都是同一份公开文本。

### 2.3 参考基线

课程 starter 包提供审计协议、划分 manifest、数据字典与格式样例，不包含任何模型；handout 明确要求自行编写 profiling、baseline 与审计代码。因此本报告中的"基线"均指下述由我们构建的参考基线，唯一由课程固定的是划分本身。

基线为 **TF-IDF + 逻辑回归**的线性文本分类器，训练两个视图：

- **word 模型**（正牌基线）：词与二元词组 TF-IDF（`ngram_range=(1,2)`，`min_df=2`，`sublinear_tf=True`），词表 12,540 维，逻辑回归 `C=4.0`、`solver='liblinear'`、`random_state=42`。
- **char 模型**（辅助信号源）：字符 3–5 gram TF-IDF（`analyzer='char_wb'`，上限 50,000 特征），同一逻辑回归配置。两个模型的差异纯粹来自文本切分方式，不含算法差异。

选择线性模型是审计需求而非性能取舍：每个预测都可以分解回具体的词与系数，这是"模型为何反对某个标签"这一证据可被人工复核的前提。

训练集上的评分采用 **5 折 out-of-fold**。若用全训练集拟合后再预测训练集自身，模型已记忆每行标签，"模型反对该标签"这一信号即失效；OOF 保证为每行打分的模型从未见过该行，这是目标 4 成立的必要条件。held-out 由全训练集拟合的模型直接预测。

| 条件                     |     n |             准确率 | spam 精确率 | spam 召回率 | spam F1 |
| ------------------------ | ----: | -----------------: | ----------: | ----------: | ------: |
| word heldout（参考基线） | 1,114 | **99.1023%** |      0.9929 |      0.9396 |  0.9655 |
| char heldout             | 1,114 |           98.7433% |      0.9927 |      0.9128 |  0.9510 |
| word OOF（train）        | 4,460 |           98.2063% |      0.9924 |      0.8729 |  0.9288 |

word 模型系数最高的 spam 特征为 `call` (+6.69)、`txt` (+6.65)、`text` (+5.54)、`free` (+4.62)、`mobile` (+4.43)、`claim` (+4.41)、`www` (+4.33)；ham 侧最负的为 `my` (−3.28)、`me` (−3.12)、`that` (−2.76)、`ok` (−2.69)。spam 侧集中于促销行为动词与渠道词，ham 侧集中于第一人称代词。这一分布本身即预示了目标 5 的结论。

**一处需要解释的反常。** held-out 的 spam 召回率（0.9396）比同分布的 OOF（0.8729）高出 6.7 个百分点。held-out 从未参与任何调参，其分数按理不应高于交叉验证估计。这一缺口在 §3.3 与 §7.2 得到解释：held-out 中混有模型在训练时已经见过的行。

### 2.4 审计方法论：检索与判定分离

全部六个目标共用同一条方法论主线：**自动信号负责把候选从 5,574 行中检索出来，判定则另走人工政策阅读**。二者的产物分开存储，被拒候选与被接受的发现一同留档。

| 目标         | 检索方法                                        | 判定依据                 |
| ------------ | ----------------------------------------------- | ------------------------ |
| 1 精确重复   | 按归一化文本（小写 + 空白折叠）分组             | 客观字符串规则，无需人工 |
| 2 近重复     | TF-IDF 余弦 ≥ 0.92 + 阈值敏感度分析            | 人工逐对 accept / reject |
| 3 跨划分泄漏 | 将目标 1、2 的证据跨划分汇总 + 行号位置检查     | 继承 1、2 的判定         |
| 4 标签错误   | word / char OOF + 近邻，要求 ≥2 个独立信号反对 | 信号 + 人工阅读原文      |
| 5 shortcut   | 仅供给手工特征的浅层探针 vs 同算法全文对照      | 标准化系数 + 性能占比    |
| 6 模糊       | 模型分歧 / 边界距离 / 邻居标签混杂（任一命中）  | 标注政策阅读             |

这一分离带来的直接后果是，检索阶段的宽严可以按主张强度分别设定（详见 §6.1），而不必为了迁就单一门槛牺牲召回或精度。

## 3. 目标 1–3：重复与跨划分泄漏

### 3.1 精确重复

精确重复按协议的公开规则定义：小写化并折叠空白后字符串完全相同。归一化后语料剩余 5,159 条唯一文本，即 415 行属于首次出现之外的重复。

| 指标             | 结果 |
| ---------------- | ---: |
| 精确重复簇       |  290 |
| 涉及行数         |  705 |
| 跨划分簇         |    5 |
| 存在标签冲突的簇 |    0 |
| 最大簇规模       |   30 |

**没有任何一个精确重复簇内部存在标签冲突**，即不存在"同一文本被标成不同标签"的情况。这说明标注工作在字面重复层面是自洽的，是一个正面的数据质量发现。

报告采用**按簇**呈现（每簇一个代表行）而非逐条列出 705 个成员。置信度与严重度分开记录：精确重复的证据是字符串恒等，可机械验证、不存在误报空间，因此 290 个簇的置信度**全部为 high**；严重度的差异由 `recommended_action` 承载——跨划分的 5 个簇走 `rebuild_split_by_cluster`（威胁评估完整性），其余 285 个走 `dedupe_by_cluster`（常规去重清理）。§8 详细说明这一口径。

### 3.2 近重复

候选检索使用 word 与 char 两套 TF-IDF 视图，主分取 `max(word_cos, char_cos)`，检索下限 0.88，协议判定阈值 0.92。为使阈值选择可审计而非隐藏调参，我们在 0.88–0.96 区间做了阈值敏感度分析，记录各阈值下的候选对数与跨划分对数变化。

0.92 阈值下得到 365 对候选。排除 17 对 heldout–heldout 组合（不影响评估完整性）后，348 对进入人工评审：

| 评审结果                        |          对数 | 其中跨划分 |
| ------------------------------- | ------------: | ---------: |
| accepted（确为近重复）          | **200** |         85 |
| rejected（判为 false positive） | **148** |          5 |
| 未决                            |             0 |         — |

被拒的 148 对可归为两类：**仅共享通用短语或寒暄而无具体内容锚点**，以及**字面高度相似但语义实质改变**。这里体现了一条刻意设定的证据标准：同一句通用短语，作为**精确**重复成立（这是客观字符串事实），作为**近**重复不成立——近重复必须能指向同一来源、同一模板或同一具体内容。

### 3.3 跨划分泄漏

将精确重复与已接受的近重复证据跨划分汇总，共 **63 个 held-out 行**可从训练集恢复（5 high / 46 medium-high / 12 medium）：

| 泄漏通道             | 影响的唯一 held-out 行 |
| -------------------- | ---------------------: |
| 精确重复跨划分       |                      5 |
| 已接受的近重复跨划分 |                     60 |
| 去重后合计           |           **63** |

即 held-out 的 **5.7%** 属于"考已经见过的题"。§7.2 量化了这部分泄漏在分数上值多少。

**行号位置未构成泄漏通道。** 独立检验显示，仅以归一化 UCI 行号为特征的探针达到 86.62% 准确率与 0.00% spam F1，与多数类基准完全一致（§5.2），排除了语料按标签排序导致行序泄漏的可能。

## 4. 目标 4：疑似标签错误

### 4.1 方法

协议对 label_error 的定义要求至少两个独立信号反对现有标签。我们据此对全部 5,574 行计算三个信号：

| 信号      | 来源                         | 独立性依据                                       |
| --------- | ---------------------------- | ------------------------------------------------ |
| word 反对 | 词/二元词组 TF-IDF 逻辑回归  | 训练行为 OOF 打分，模型未见过该行                |
| char 反对 | 字符 3–5 gram 逻辑回归      | 独立特征空间，可捕捉`fr33`、`w1n` 类变形写法 |
| 近邻反对  | 字符 TF-IDF 最近邻标签一致性 | 局部标签结构，独立于任何分类决策边界             |

候选门槛严格取协议定义（**至少两个信号反对**），得到 70 个候选。近邻窗口**严格排除归一化文本相同的邻居**：若不排除，一条数据会被自己的重复副本"确认"标签，证据将自我污染（详见 §9 困难 3）。

置信度由可复现的规则决定，而非事后主观赋值：

> high = 反对强度 ≥ 0.95，或三个信号全部反对
> medium = 两个信号反对且强度 < 0.95
> low = 不足两个自动信号，主张仅由政策阅读支撑

### 4.2 发现

最终提交 **15 条疑似标签错误**（5 high / 9 medium / 1 low），全部为公开标签 spam、主张改为 ham 的方向。其中 13 条位于训练集（进入 overlay 实验），2 条位于 held-out（**仅标记，不修改标签**，遵守 handout 硬约束）。

| ID    | 划分    | 置信度 | 反对强度 | 信号数 | 判读                                 |
| ----- | ------- | ------ | -------: | ------ | ------------------------------------ |
| T2378 | train   | high   |    0.992 | 3/3    | 观察类开车笑话，无任何招揽           |
| H0143 | heldout | high   |    0.982 | 2/3    | 纹身笑话，仅标记不改                 |
| T0059 | train   | high   |    0.965 | 2/3    | "Divorce Barbie" 段子                |
| T2232 | train   | high   |    0.963 | 2/3    | 小孩与青少年对比笑话                 |
| H0896 | heldout | high   |    0.916 | 3/3    | 受害者询问"中奖后怎么办"，仅标记不改 |

其中浮现出一个值得单独命名的政策模式：**T1798、T4296、T1174 是在讨论、引用或抱怨垃圾短信，而非发送垃圾短信。** 它们字面高度像 spam（复述了促销话术），语义上是普通个人消息。T1174 仅有单一信号支持，是全部主张中最弱的一条，如实标记为 low。

### 4.3 被拒候选与 false-positive 防护

在人工阅读的 45 个高排名候选中，**31 个被裁定为 `keep_but_flag`**——即模型判断有误，公开的 spam 标签成立。被拒模式集中于四类：

| 模式           | 例                  | 击败模型判断的证据                           |
| -------------- | ------------------- | -------------------------------------------- |
| 尾部披露资费   | T2755, T3105, T0006 | `one gbp/sms`、`cst std ntwk chg £1.50` |
| 回复关键词机制 | H0814, T3202        | `Send A, B or C`、短码                     |
| 成人内容招揽   | T1505, T1926        | `Txt XXX SLO(4msgs)` 类指令                |
| 合法商业广播   | T2229, T2273        | Burger King、Interflora 促销                 |

**T2755 说明了为何拒绝环节不能自动化**：其反对强度 0.953 已越过 high 门槛，但文本结尾附有 `one gbp/sms` 的资费披露——不实际阅读原文，就会把它误判为一条 high 置信度的标签错误。这类案例正是 hidden audit key 中 false-positive trap 所针对的形态。

全部 80 条经过裁定的候选连同裁定理由记录于 `results/adjudication_memo_b.csv`，裁定分布为：`keep_but_flag` 49、`should_fix` 15、`ambiguous_policy_case` 14、`false_positive_audit_finding` 2。

## 5. 目标 5：Shortcut 特征

### 5.1 方法

浅层探针**只接收 12 个派生数值特征，绝不接收原始词序**：字符数、词数、平均词长、数字个数、数字比例、大写比例、感叹号数、货币标记数、是否含 URL、是否含电话号形式串、促销词计数，以及独立测试的归一化行号位置。数值特征经标准化后输入逻辑回归。

对照组的设计是本节结论成立的关键：**全文对照使用同一算法（逻辑回归）**，唯一变量是特征集合。若对照组换用其他模型族，性能差异就无法归因于特征而只能归因于模型，整个论证失效。全文对照的结果直接读取冻结的共享 held-out 预测，不重新训练。

### 5.2 结果

| 特征集                      |           准确率 | spam 精确率 | spam 召回率 |          spam F1 |
| --------------------------- | ---------------: | ----------: | ----------: | ---------------: |
| 多数类（恒输出 ham）        |           86.62% |       0.00% |       0.00% |            0.00% |
| **仅行号位置**        |           86.62% |       0.00% |       0.00% |            0.00% |
| 仅形状（长度/大写/叹号）    |           89.50% |      70.00% |      37.58% |           48.91% |
| **仅联系方式 + 促销** | **98.11%** |      95.07% |      90.60% | **92.78%** |
| 全部浅层特征                |           98.20% |      95.10% |      91.28% |           93.15% |
| 全部浅层 + 行号位置         |           98.20% |      95.10% |      91.28% |           93.15% |
| 遮蔽 15 个促销词后的全文    |           98.92% |      97.90% |      93.96% |           95.89% |
| 完整全文（参考基线）        |           99.10% |      99.29% |      93.96% |           96.55% |

**主要结论：仅凭联系方式与促销线索即可达到 98.11% 准确率与 92.78% spam F1，与完整词模型的 99.10% / 96.55% 相差约 1 个百分点。** 数据集的绝大部分可分性由浅层表面线索承载，模型不必理解句意即可接近上限。

协议规定的 70% 准确率警戒阈值在本数据集上不具判别力——多数类分类器已达 86.62% 而完全不检出 spam。真正有信息量的判据是浅层探针与完整模型之间的差距被压缩到约 1 个百分点。

**行号位置单独毫无预测力**（准确率与 spam F1 均与多数类基准完全相同），这是一个干净的反面结果，排除了行序 shortcut 的可能。

### 5.3 具体是哪些线索

标准化系数给出了直接答案（正值指向 spam）：`digit_count` (+2.566)、`promo_token_count` (+1.723)、`avg_token_length` (+0.760)、`has_url` (+0.678)、`currency_count` (+0.642)、`exclamation_count` (+0.336)。因输入已标准化，同一拟合内的系数量级可直接比较。

### 5.4 解读与限度

数字、价格、URL、电话号与促销措辞是**合法的** spam 证据，其预测力本身不构成泄漏。它们之所以被判定为 shortcut，是因为它们**可以在不理解句意的情况下生效，并在分布偏移下失效**：普通的价格讨论会触发假正例（H0946：一条正当的房租讨论因含十一个数字与三个货币标记被推向 spam），而缺少常见数字标记的招揽会被漏掉（H0044：铃声俱乐部招揽几乎不含数字与联系线索，浅层探针漏检而文本模型检出）。

系数描述关联而非因果。本节的 8 条示例行是压力测试案例，不构成修改 held-out 标签的请求；其中涉及标注政策的行（如 H0143、H0896）已移交目标 4 联合裁定，未在本节重复计为 shortcut 证据。

## 6. 目标 6：模糊样本与分数上限

### 6.1 检索：三个问题，命中其一即入池

我们要寻找的是"标 ham 与标 spam 都讲得通"的消息。这类性质无法用规则直接判定，因此先用三个彼此独立的问题捞取可疑对象，再逐条人工阅读原文：

1. **两个模型是否给出相反判断？** 词模型与字符模型对同一条消息一个判 ham、一个判 spam。→ 45 条
2. **模型自身是否犹豫？** 预测概率落在 0.32–0.68 区间。→ 101 条
3. **邻居的标签是否混杂？** 取文本最相似的一批消息，若其标签接近对半分，说明**标注者在处理这类消息时本身就不一致**。→ 100 条

三个问题合计触发 246 次，去重后得到 **177 条候选**（train 144 / heldout 33），说明三者确实捕捉到部分不同的现象，而非同一件事的三种问法。

第三个问题值得单独说明：前两个问的都是**模型怎么想**，只有第三个问的是**当初标注的人怎么做的**。邻居标签对半分是标注指南本身存在歧义的直接痕迹，与模型强弱无关。

**为何此处用"或"，目标 4 却用"且"？** 差别在于两边主张的强度不同。"这条标错了"是在断言标注**有误**，主张强，因此检索阶段就必须卡严，宁可漏检；"这条两说"只是在说这条**值得阅读**，捞宽无妨，因为后续每条都会经人工审阅。检索门槛应当匹配主张强度，而非全局统一。

候选表附带一个 `ambiguity_score` 排序分，将上述三个问题加权合成，**仅用于决定人工阅读的先后顺序，不参与任何判定**。

### 6.2 判定：模糊 ≠ 低置信度

"模糊"仅保留给一种情况：**在不知道发送方身份、用户是否曾订阅、是否属于某个营销活动的前提下，判 ham 与判 spam 都成立。**

这与"模型没把握"是两回事。模型没把握可能仅仅是模型不够强；而模糊指的是**信息本身不足以做出判断**，换用更强的模型同样判不了。若把模型犹豫的样本直接上报为模糊，测量的对象就从数据集变成了分类器。

支持这一区分的直接证据是：最终认定的 14 条模糊发现中，**10 条来自标签错误候选池而非边界候选池**。其中最强的两条 T2139（反对强度 0.987、3/3 信号）与 T2710（0.951、3/3）**模型毫不犹豫**，但它们仍然是标注政策问题。两条判据正交。

177 条候选全部经人工阅读，最终仅 **14 条**被认定为模糊（7.9%）。其余绝大多数读后归入"模型确实拿不准，但标注本身无争议"，不构成数据集缺陷。

### 6.3 发现

提交 14 条模糊（13 medium / 1 low），典型的政策边界包括：

- **070 号码对（T2710 / T3390）**：英国 070 号段既是合法的个人转接段，也常被付费重定向服务滥用；两条均无资费披露，按同一标准判为模糊。
- **可能已订阅的招揽（T2847 俱乐部票、T3981 UNICEF 捐款）**：会员名单与慈善豁免是真实存在的政策 carve-out。它们与 T2229 / T2273（无任何同意背景的纯商业广播，被拒为成立的 spam）构成对照，划出了这条边界的两侧。
- **背景不可恢复（T0774 文本截断、H0550 语料残留）**：文本已不足以支撑判定，这本身就是关于语料质量的发现。

### 6.4 刻意保留的 false positive

两条以 low 置信度提交、并明确标注为被拒候选，用以证明"预测不确定"既不等于标签错误，也不等于语义模糊：

| ID    | 文本                     | 为何不是发现                                                                      |
| ----- | ------------------------ | --------------------------------------------------------------------------------- |
| H0935 | "I liked the new mobile" | 概率骑在 0.5 附近、邻居标签混杂，但内容就是普通 ham                               |
| T0470 | "Waiting for your call." | word 概率 0.689 源于与 CTA 类 spam 的词面重叠，但**没有任何邻居反对该标签** |

保留这两条是有代价的——它们会稀释发现集合的表观精度——但正是这种代价使得置信度分布具有可解释性。

### 6.5 模糊样本如何限制分数上限

handout 对目标 6 的提问有两半，后半是"它们如何限制 score ceiling"。以下正面回答。

先给出立论依据。参考基线在 1,114 个 held-out 行上共犯 **10 个错误**，按审计状态分类：

| 状态             | 数量 | 行                                              |
| ---------------- | ---: | ----------------------------------------------- |
| 真实模型错误     |    7 | H0287, H0318, H0330, H0461, H0814, H0903, H0927 |
| 疑似公开标签错误 |    2 | H0143, H0896                                    |
| 模糊政策案例     |    1 | H0550                                           |

**10 个错误中有 3 个是数据问题而非模型失败。** 这一事实通过三个层次限制分数上限。

**其一，存在模型无法逾越的误差地板。** 一条真正模糊的样本按定义没有唯一正确标签；无论模型预测哪一边，它被评分所对照的那个标签都只是一种任意约定。这类行构成**不可约误差**。据此可以计算上界——即使一个完美模型修正了全部 7 个真实错误，上表中的 3 个数据问题行仍会被判错：

```
上界 = (1114 − 3) / 1114 = 99.7307%
```

在数据保持现状的前提下，任何模型的 held-out 准确率都不可能超过 **99.73%**。

**其二，剩余提升空间与争议区已处于同一量级。**

|                                     |       行数 | 占 held-out |
| ----------------------------------- | ---------: | ----------: |
| 当前 99.1023% 之上的全部剩余空间    |          7 |       0.63% |
| 标注存在争议的行（模糊 + 疑似标错） |          3 |       0.27% |
| 二者之比                            | ≈ 2.3 : 1 |             |

这意味着**任何小于约 0.3 个百分点的性能提升，都无法与"重新裁定几条标注政策"区分开**。该基准已进入噪声主导区：分数小数点后两位测量的不再是模型能力，而是标注约定。§7.2 的裁定感知评估从实测侧印证了这一点——仅将 4 行移出评分，头条数字即从 99.1023% 变为 99.3694%。

**其三，训练侧的约定固化。** 14 条模糊发现中有 13 条位于训练集。它们不直接影响 held-out 分数，但会**把一个任意的标注约定教给模型**（例如"070 号码一律算 spam"）。模型学到的是这个约定而非可泛化的规律；而当评估集沿用同一约定时，这份偏差反而会被奖励——§7.3 中"修正训练标签反而掉分"正是这一机制的直接体现。

**小结**：模糊样本以三种方式限制分数上限——设定 99.73% 的硬上界、把剩余提升空间压缩到与争议区同量级从而使小幅改进不可解释、并经由训练集把任意约定固化进模型。在该数据集上继续追求更高分数，衡量的已不再是模型质量。

## 7. 数据干预对比

handout 要求比较至少两种数据干预。本报告实施了**九种**，覆盖全部六个审计目标，并由两套相互独立的实现交叉验证（§7.5）。

两条贯穿始终的约束：**标准工作表永不修改**（所有干预以 overlay 或派生副本表达），**held-out 标签永不更改**。

### 7.1 干预矩阵

按"改动了数据的哪一部分"分类，而非按审计目标分类——这一分类轴能直接解释各干预升分或降分的原因。

| 干预                   | 改动对象           | 规模           | 重训 |             准确率 |        spam 召回率 |
| ---------------------- | ------------------ | -------------- | ---- | -----------------: | -----------------: |
| **参考基线**     | —                 | —             | —   |           99.1023% |           93.9597% |
| *评估侧（模型冻结）* |                    |                |      |                    |                    |
| 泄漏感知评估           | 移出评估行         | −63           | 否   |           99.0485% | **90.9091%** |
| 裁定感知评估           | 移出评估行         | −4            | 否   | **99.3694%** | **95.8621%** |
| *训练侧（须重训）*   |                    |                |      |                    |                    |
| 训练标签 overlay       | 改训练标签         | 13 条重标      | 是   |           98.9228% |           92.6174% |
| 移除泄漏伙伴行         | 删训练行           | −71           | 是   |           99.0126% |           92.6174% |
| 排除模糊训练行         | 删训练行           | −12           | 是   |           99.0126% |           92.6174% |
| 按簇去重               | 删训练行           | −417          | 是   |           98.8330% |           92.6174% |
| 促销词遮蔽（15 词）    | 改文本             | 全量           | 是   |           98.9228% |           93.9597% |
| 线索遮蔽（全类别）     | 改文本             | 全量           | 是   |           98.7433% |           93.9597% |
| *组合*               |                    |                |      |                    |                    |
| 组合训练视图           | 上述训练侧全部叠加 | 4,460 → 3,982 | 是   |           98.4740% |           91.9463% |

**阅读这张表的关键：全部干预对准确率的影响都在 0.7 个百分点以内，而 spam 召回率的分布从 90.91% 横跨到 95.86%，跨度是准确率的七倍。** §7.6 展开这一观察的方法论含义。

### 7.2 评估侧干预：不动模型，只改测量

这两个干预共享同一机制——**冻结基线预测，只改变哪些行参与评分**——但方向相反，恰好构成一组对照。

#### 泄漏感知评估（目标 3）

将 §3.3 中可从训练集恢复的 held-out 行移出评分。设计为分级形式，以区分两种泄漏通道：

| 条件                        |     移出行数 |     n |   准确率 |        spam 召回率 |
| --------------------------- | -----------: | ----: | -------: | -----------------: |
| 完整 held-out（参考基线）   |            0 | 1,114 | 99.1023% |           93.9597% |
| 仅移出精确重复泄漏          |            5 | 1,109 | 99.0983% |           93.7931% |
| 移出精确 + 已接受近重复泄漏 | **63** | 1,051 | 99.0485% | **90.9091%** |

**分级揭示了一个从总数看不出来的事实：真正的伤害几乎全部来自近重复，而非精确重复。** 仅移出 5 条精确泄漏，spam 召回率只降 0.17 个百分点；再移出 58 条近重复泄漏，直接降 2.88 个百分点。若审计止步于精确字符串匹配，就会得出"泄漏影响可忽略"的错误结论——§3.2 中 348 对人工评审的工作量，其必要性正由这张表证明。

被移出的 63 行中，基线**全部预测正确**（63/63），因为模型在训练时见过它们；其中 50 行是 spam，正是它们把真实的 90.91% spam 召回率抬高到 93.96%。这是全部干预中影响最大的单项。

#### 裁定感知评估（目标 4 / 6）

移出审计判定为**模糊**（不存在唯一正确答案）或**疑似标错**（正按我们认为有误的标签接受评分）的 held-out 行——两类都不构成对模型的公平测试。

| 条件                            |    移出行数 |     n |             准确率 |        spam 召回率 |
| ------------------------------- | ----------: | ----: | -----------------: | -----------------: |
| 完整 held-out（参考基线）       |           0 | 1,114 |           99.1023% |           93.9597% |
| 移出模糊行（H0030 / H0550）     |           2 | 1,112 |           99.1906% |           94.5578% |
| 再移出疑似标错（H0143 / H0896） | **4** | 1,110 | **99.3694%** | **95.8621%** |

held-out 上另有一条标为 ambiguous 的 **H0935 未被移出**：它是 §6.4 中刻意保留的 false positive，已被裁定为普通无争议样本。既然自行判定它不模糊，就不能再以模糊为由把它移出评分。这一处理是判定一致性的检验点。

**这个干预的一半按设计不构成独立证据，需要明确声明。** 被移出的 4 行中有 3 行是基线原本就答错的，但两部分的性质截然不同：

| 移出的行                     | 基线答错 | 是否与检索方式循环 |
| ---------------------------- | -------: | ------------------ |
| 疑似标签错误（H0143, H0896） |    2 / 2 | **是**       |
| 模糊（H0030, H0550）         |    1 / 2 | 否                 |

标签错误那两行是靠"≥2 个模型信号反对该标签"检索出来的，而在 held-out 上，"模型反对这个标签"与"模型答错了"本就是同一件事。移出它们必然涨分——这不是实验结果，而是同一事实的换算。**因此准确率从 99.19% 上升到 99.37% 的那一段不构成任何证据，本报告不据此立论。**

**真正有信息量的是模糊那两行：模型答对了其中一条（H0030）。** 这批行由人工阅读原文做政策判断挑出，与模型是否答对无关。结果表明模糊行**并非"模型失败样本"的同义词**，从数据侧支持了 §6.2 的立论：模糊是标注层面的性质，不是分类器的状态。此处样本量 n = 2，量化的是机制而非效应量；§6.5 的上界论证依赖的是那个硬上界，而非这 +0.09 个百分点。

### 7.3 训练侧干预：改动数据后重训

#### 训练标签 overlay（目标 4）

将 §4 认定的 13 条训练标签修正以 overlay 文件形式应用（不覆盖公开标签），重训 word 基线（SEED = 42），在完整 held-out 上评估：

| 条件                 |   准确率 | spam 召回率 |
| -------------------- | -------: | ----------: |
| 公开标签（参考基线） | 99.1023% |    93.9597% |
| 应用 overlay         | 98.9228% |    92.6174% |

**分数下降，但这不表示干预方向错误。** 我们修正的是"公开标签把若干笑话与转述标记为 spam"，而 held-out 仍按**同一套公开惯例**评分——纠正训练集反而使模型在沿用旧惯例的评估集上失分。这恰恰说明：**当评估集本身带有标注噪声时，基准分数会惩罚朝向正确方向的数据修正。** 这是本报告对"分数即质量"这一默认假设最直接的反例。

#### 促销词遮蔽 / 线索遮蔽（目标 5）

§5 已证明模型高度依赖表面促销线索。我们施加两种遮蔽范围，构成一组强度梯度：

| 条件                 | 遮蔽范围                                   |   准确率 | spam 精确率 | spam 召回率 | 假正例数 |
| -------------------- | ------------------------------------------ | -------: | ----------: | ----------: | -------: |
| 完整全文（参考基线） | —                                         | 99.1023% |      0.9929 |      0.9396 |        1 |
| 促销词遮蔽           | 固定的 15 个促销词                         | 98.9228% |      0.9790 |      0.9396 |        3 |
| 线索遮蔽             | URL + 电话 + 货币/资费 + 数字串 + 促销词库 | 98.7433% |      0.9655 |      0.9396 |        5 |

**三个条件的 spam 召回率完全相同（0.9396），差异全部落在精确率上（0.9929 → 0.9790 → 0.9655）。**

这个结果比任一单独条件都更有说明力：遮蔽越激进，模型**并没有漏掉更多 spam**，只是**误判了更多 ham**。这说明表面线索并非识别 spam 的必要条件——去掉一批，模型平滑地退守到另一批（短码、格式等结构性线索）上。**所谓 shortcut 不是一根独木桥，而是一整片高度冗余的表面线索网络。** 这也解释了 §5.2 中仅凭"联系方式 + 促销"特征即可达到 98.11% 的现象。

#### 三种删除训练行的干预

| 干预           | 删除对象                     |      训练行数 |   准确率 | spam 召回率 |
| -------------- | ---------------------------- | ------------: | -------: | ----------: |
| 参考基线       | —                           |         4,460 | 99.1023% |    93.9597% |
| 移除泄漏伙伴行 | 与 held-out 构成重复的训练行 |  4,389 (−71) | 99.0126% |    92.6174% |
| 排除模糊训练行 | 裁定为模糊政策案例的训练行   |  4,448 (−12) | 99.0126% |    92.6174% |
| 按簇去重       | 每个重复簇仅保留最早一行     | 4,043 (−417) | 98.8330% |    92.6174% |

**四个条件的 spam 召回率全部落在 92.6174%**，而删除规模从 12 行跨越到 417 行。删掉 417 行（近 10% 的训练数据）与删掉 12 行的效果相同，说明**被删除的绝大多数是冗余副本，模型本就未从中获得额外信息**——这从反面印证了 §3.1 的重复发现：那 415 行多余副本确实是多余的。

值得记录的是，移除泄漏伙伴行（−71）与排除模糊训练行（−12）产生了逐格相同的混淆矩阵（tn = 965, fp = 0, fn = 11, tp = 138）。删除两批完全不同的训练行却得到同一组 held-out 判决，进一步支持上述解读：这两批行都不携带决定性的训练信号，模型的决策边界不依赖它们。

### 7.4 组合干预

将训练侧全部干预叠加（移除泄漏伙伴 + 按簇去重 + 排除模糊 + 剩余标签 overlay + 线索遮蔽），训练集从 4,460 降至 **3,982 行**：

| 条件         | 训练行数 |             准确率 |        spam 召回率 |
| ------------ | -------: | -----------------: | -----------------: |
| 参考基线     |    4,460 |           99.1023% |           93.9597% |
| 组合训练视图 |    3,982 | **98.4740%** | **91.9463%** |

这是全部干预中降幅最大的（准确率 −0.63 pt，spam 召回 −2.01 pt），但**它不构成"数据变差了"的证据**。这是一个保守的压力测试，而非拟议的替代数据集。在一个评估集本身含泄漏、含标注噪声、并奖励表面线索的基准上，**训练数据修得越干净、分数越低是可以预期的**——这正是 §7.6 观察三的极端形态。其价值在于给出数据质量问题的累积量级：每个问题单独看都很温和，合并起来则明显削弱了"头条指标代表干净、独立的泛化能力"这一判断。

### 7.5 交叉验证

本节结果由两套彼此独立的实现产出：面向单一目标的专用脚本（`label_overlay_experiment.py`、`intervention_leakage_eval.py`、`intervention_ambiguity_eval.py`）与统一批处理脚本 `run_impact_interventions.py`。三个共有条件的结果逐位一致：

| 条件             | 两套实现                                   |
| ---------------- | ------------------------------------------ |
| 参考基线         | 准确率 0.991023 / spam 召回 0.939597，一致 |
| 泄漏感知评估     | 准确率 0.990485 / spam 召回 0.909091，一致 |
| 训练标签 overlay | 准确率 0.989228 / spam 召回 0.926174，一致 |

独立实现给出相同数字，是对流水线正确性的实质性检验，强于单一实现的自我验证。

### 7.6 综合解读

| 干预               |         准确率变化 |       spam 召回变化 | spam 精确率变化 |
| ------------------ | -----------------: | ------------------: | --------------: |
| 泄漏感知评估       |          −0.05 pt | **−3.05 pt** |       −0.39 pt |
| 裁定感知评估       |           +0.27 pt |            +1.90 pt |       −0.01 pt |
| 训练标签 overlay   |          −0.18 pt |           −1.34 pt |       −0.01 pt |
| 促销词遮蔽         |          −0.18 pt |       **±0** |       −1.39 pt |
| 线索遮蔽           |          −0.36 pt |       **±0** |       −2.74 pt |
| 删除训练行（三种） | −0.09 ~ −0.27 pt |           −1.34 pt |  +0.7 ~ +1.4 pt |
| 组合训练视图       |          −0.63 pt |           −2.01 pt |       −2.81 pt |

**观察一：泄漏相当于白送了 63 道已经考过的题。** 被移出的 63 个 held-out 行基线全部答对，其中 50 个是 spam，正是它们把真实的 90.91% spam 召回率抬到 93.96%。这是所有干预中影响最大的单项。

**观察二：准确率会把机制不同的数据问题压缩成同一个假象。** 九种干预对准确率的影响全部挤在 0.7 个百分点以内，而 spam 召回率跨越 4.95 个百分点，且模式各不相同——遮蔽类只动精确率，删行类只动召回率，裁定感知评估的方向甚至相反。**若只报告准确率，这九种数据问题看起来完全一样（都"几乎无影响"），而真相是它们的作用机制彼此无关。** 这是本报告最核心的方法论结论：在 13.4% 的类别不平衡下，准确率不是一个中性的汇总指标，而是一个会系统性抹平差异的指标。

**观察三：基准分数同时被两个相反方向扭曲。** 泄漏感知评估表明它**虚高**（有送分行）；训练标签 overlay 与组合视图表明它会**惩罚正确的数据修正**（评估集沿用同一套有问题的惯例）；裁定感知评估的上升则大部分属于记账口径而非新增证据（§7.2）。综合起来：99.10% 既不是模型真实能力的上界，也不是一个"把数据修干净就能单调改善"的目标——**它是若干方向相反的数据缺陷相互抵消之后的结果。

### 8.1 两个提交文件的分工

| --- | --- | ---: | ---: | ---: | ---: |
| 1 | `exact_duplicate` | 290（每簇一行） | 290 | — | — |
| 2 | `near_duplicate` | 278 | — | 127 | 151 |
| 3 | `leakage` | 63 | 5 | 58 | — |
| 4 | `label_error` | 15 | 5 | 9 | 1 |
| 5 | `shortcut` | 8 | 3 | 5 | — |
| 6 | `ambiguous` | 16 | — | 13 | 3 |
| | **合计** | **670** | **303** | **212** | **155** |

协议的三条硬约束全部满足：行数 670 ≥ 35、六类全覆盖、high 占比 **45.22% ≤ 55%**。

两点读表说明。其一，`exact_duplicate` 按簇报告，290 行对应 290 个簇（覆盖 705 个成员行），而非逐条列出成员。其二，`ambiguous` 的 16 行中有 **14 条是真正的模糊发现**（13 medium + 1 low），另外 2 行（H0935、T0470）是 §6.4 中刻意保留的 false positive，以 low 提交并标记 `reject_audit_finding`——它们出现在提交文件中是为了让校准可被检验，不是主张这两行有问题。

`adjudication_memo.csv` 共 **444 行**，归并三处人工评审记录，并映射到协议规定的四个裁定类别：

| 来源 | 行数 | 覆盖目标 |
| --- | ---: | --- |
| 标签错误与模糊裁定备忘 | 80 | 4、6 |
| 近重复逐对评审 | 348 | 2 |
| 捷径候选评审 | 16 | 5 |

裁定分布：`should_fix` 215、`false_positive_audit_finding` **158**、`keep_but_flag` 57、`ambiguous_policy_case` 14。

### 8.2 置信度口径

置信度回答的是**"这一行确实属于所标 issue_type 的把握有多大"**，严重度则由 `recommended_action` 单独承载。二者分离是有意为之：一个跨划分的重复簇与一个簇内重复簇，证据强度同样确定（都是字符串恒等），但前者威胁评估完整性、后者只是常规清理——这个差别应当体现在建议动作上，而不是通过压低置信度来表达。

| 置信度 | 判据 | 典型来源 |
| --- | --- | --- |
| high | 可机械验证的证据，或反对强度 ≥ 0.95 / 三信号全反对 | 精确重复簇、精确重复支撑的泄漏行 |
| medium | 客观证据但需要判断介入 | 已接受的近重复对、两信号标签错误、模糊政策案例 |
| low | 最弱的保留主张，含刻意保留的 false positive | 单信号标签错误、通用短语类近重复、对照案例 |

按此口径，290 个精确重复簇全部为 high——它们是字符串恒等，不存在误报空间。high 占比因此达到 45.22%，仍在协议上限之内。

### 8.3 校准的代价

协议的 55% 上限是一条有意设置的校准约束：它使"全部标为 high"在形式上不可行，迫使发现集合表达出内部的强弱层次。但真正体现校准的不是这个比例，而是我们主动做的三件降低表观发现数量的事：

1. **把 148 对近重复候选判为 false positive**——它们仅共享通用短语，或字面相似但语义实质改变；
2. **把 31 条标签错误候选退回为 `keep_but_flag`**——模型判断有误，公开的 spam 标签成立；
3. **把 2 条明确判定为非问题的行以 low 置信度提交为对照**（H0935、T0470），用以证明"预测不确定"不构成发现。

这三项合计 181 个候选被排除或降级，全部连同理由留档于 `adjudication_memo.csv`。依据是 handout 第 IV 节的明确要求：一长串弱证据的 high 置信度发现，其价值低于一个能清楚解释不确定性与被拒候选的排序列表。

需要说明取舍的另一面：精确重复与近重复是可验证的客观事实，误报风险极低，因此我们选择完整报告而非抽样。这提高了对已知问题的覆盖，同时不以牺牲置信度校准为代价——670 行中真正做出强主张（high）的只有 303 行，且每一行都有可机械复核的证据。

## 9. 困难与解决方案

### 困难 1：让包含人工判断的流水线完整可复现

**问题。** 近重复评审涉及 348 对候选的人工 accept / reject 判断，这些判断必须能与自动检索出的候选集合稳定对应。若二者的对应关系依赖于任何运行时产生的临时标识，流水线就无法复现——重跑一次，判断与候选就会错位，下游的泄漏结论随之改变。这是本项目最实质的工程挑战，因为它同时涉及三个层次：

| 层次 | 风险来源                                         | 采取的设计                                                                   |
| ---- | ------------------------------------------------ | ---------------------------------------------------------------------------- |
| 1    | 候选对的标识若来自遍历或插入顺序，会随实现漂移   | 先按`(id_a, id_b)` 字典序排序再编号，标识与遍历顺序解耦                    |
| 2    | 人工判断若嵌入脚本常量，等于绑死在会漂移的标识上 | 判断外置为独立数据文件，按`pair_key = "\|".join(sorted([id_a, id_b]))` 索引 |
| 3    | **候选集合本身必须确定**                   | 由最近邻 top-K 检索改为全量成对余弦阈值扫描                                  |

第 3 层最不易察觉。即使前两层已经处理妥当，基于 `NearestNeighbors` 的 top-K 检索在余弦值并列的边界上返回哪一个邻居是不确定的，两次运行的候选**集合**可能相差若干对（实测为 `T1545|T3892`、`T3609|T3892` 两对）。改为对全部样本计算成对余弦、按阈值取上三角后，候选集合不再依赖任何遍历顺序。

**结果与推广。** 三个脚本按序连续执行两遍，产物逐字节相同。可推广的经验是：**人工判断是数据，不是代码**——将其写入脚本常量，就等于把它绑定到一个随实现漂移的临时标识上。同样的并列问题也存在于报告展示表的 top-8 截取中（大量 `max_cosine = 1.0000`），一并以 `(id_a, id_b)` 作为二级排序键处理。

### 困难 2：区分"语义模糊"与"模型不确定"

**问题。** 目标 6 最省力的做法是把决策边界附近的样本全部标为模糊。但这样测量的是分类器的确定性，而非标注指南的清晰度，会使目标 6 退化为目标 4 的低置信度版本。

**解决。** 在方法层面把检索与判定彻底分离：三个自动问题只负责捞取候选（§6.1），判定另走人工政策阅读，并为"模糊"设定一个不涉及模型的定义（§6.2）。在证据层面用数据验证这一区分是否真的成立：最终 14 条模糊发现中 10 条来自"模型强烈反对"的候选池，其中两条最强的行模型毫不犹豫——若二者是同一现象，这个分布不会出现。最后刻意保留 2 条 false positive（H0935、T0470）作为对照，明确记录"概率骑在 0.5"不足以构成发现。

### 困难 3：防止近邻证据自我污染

**问题。** 目标 4 的第三个信号是近邻标签一致性。但该数据集含 415 行重复副本，若近邻窗口不加限制，一条消息会把自己的精确副本取为最近邻，从而"确认"自己的标签。这会使第三个信号与第一、二个信号的独立性失效，而协议要求的正是**独立**信号。

**解决。** 近邻窗口严格排除归一化文本与目标行相同的所有邻居；同时将检索深度从 9 提高到 20，以保证排除副本后仍有足够大的有效窗口。这里存在一个明确的取舍：排除后部分行的有效邻居数减少，第三个信号对它们不可用，这些行因此只能拿到两个信号——我们接受这一召回损失，因为独立性是协议门槛的前提，不可让渡。

### 困难 4：避免用模型信号检索出的证据去证明模型更好

**问题。** 裁定感知评估（§7.2）将审计判定有疑问的 held-out 行移出评分。但目标 4 的候选正是靠"模型信号反对该标签"检索出来的，而在 held-out 上，"模型反对该标签"与"模型答错了"是同一件事。直接报告移出后的分数提升，等于用模型自己的错误来证明模型其实更强——一个完整的循环论证。

**解决。** 将干预拆成两级并分别报告（模糊行 / 疑似标错行），在报告中显式声明后者不构成独立证据，并在结论中不予采信。真正被采信的是模糊那一级：这批行由人工政策阅读挑出，与模型是否答对无关，而模型答对了其中一条——这个结果反过来验证了 §6.2 的立论。同样的自律也体现在 H0935 的处理上：我们自行判定它不模糊，因此不能又以模糊为由把它移出评分，尽管移出会使分数更好看。

## 10. 结论与建议

**结论一：数据质量改变了评估结果的含义。** 99.10% 的头条分数同时受到三重影响——跨划分泄漏（5.7% 的 held-out 行模型已见过）、浅层 shortcut（不需理解语义即可达到 98.11%）、以及标注噪声（10 个 held-out 错误中 3 个是数据问题）。该分数不是模型真实能力的度量，而是若干方向相反的数据缺陷相互抵消后的结果。

**结论二：该基准已进入噪声主导区。** 数据现状下的准确率上界为 99.73%，当前分数之上的剩余空间（7 行）与标注争议区（3 行）仅差约一倍。小于 0.3 个百分点的改进无法与重新裁定几条标注政策相区分。

**结论三：指标选择本身是一个数据质量问题。** 九种干预在准确率上几乎无差别，在 spam 召回率上跨度达 4.95 个百分点且机制互不相同。在 13.4% 的类别不平衡下，仅报告准确率会系统性地掩盖数据缺陷之间的差异。

**结论四：判断比标记更重要。** 我们在每一类目标中都区分了强发现、不确定发现与 false positive，并把全部裁定连同理由留档于 `adjudication_memo.csv`（444 条，其中 158 条为 `false_positive_audit_finding`）。提交的 670 行发现中，做出强主张的只有 303 行，且每一行都有可机械复核的证据。发现集合的价值在于其置信度分布可被解释，而非其长度。

**改进建议：**

1. **划分层面**——跨划分重复应在评估前按簇去重，或按重复簇重建 train / heldout 划分，使同一模板的所有变体落入同一侧。
2. **标注层面**——标注指南应显式定义几类反复出现的边界：用户已订阅的商业消息、慈善募捐、英国 070 号段、以及"讨论或转述垃圾短信"的个人消息。这四类占据了本次审计中绝大多数政策争议。
3. **报告层面**——报告该数据集上的性能时应同时给出 spam 召回率与 spam F1，并附带泄漏感知评估下的敏感度结果，而非仅报告单一准确率。

## 11. AI 使用声明

本项目在 AI 工具辅助下完成候选检索脚本的起草、证据表生成与初步政策判读；所有最终审计判断由团队成员逐条人工复核，并对结论文本负责。

具体分工为：AI 参与信号脚本编写、候选证据表生成与初稿政策判读的草拟；人工负责逐条阅读原始文本、做出 accept / reject / relabel 的最终裁定、并校验报告中的所有数字与脚本实测输出一致。所有 AI 产出的判断在采纳前均对照原文验证；被拒的候选与理由一并留档于 `results/adjudication_memo_b.csv`。

**本项目未使用任何伪造的双人标注一致性统计或 Cohen's kappa。** 凡存在不确定的地方，均如实标注置信度并记录被拒候选。数据集来源为公开的 UCI SMS Spam Collection，代码依赖版本见 `requirements.txt` 与 `README.md`。
